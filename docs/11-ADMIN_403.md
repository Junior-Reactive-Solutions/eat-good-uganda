# 11 — Returning a Real 403 at `/admin` on the Customer Host

## Goal

When anyone — a scraper, a curious user, a security scanner — requests `https://eatgooduganda.com/admin` (or any path under it), the response must be a genuine HTTP `403 Forbidden` with no response body beyond a minimal plaintext message. It must not be:

- A 200 OK rendering a fake "Access Denied" page
- A client-side React route showing a "403" component (which still serves 200 at the HTTP layer)
- A redirect

Reasons:

- A real 403 tells security scanners and automated tools that the path is not a legitimate surface to probe, reducing noise in our logs.
- It is honest. The path is forbidden, so say so.
- It prevents React Router or a future catch-all handler from accidentally rendering application chrome around an error page.
- It fails closed: even if someone flips a feature flag in the future, the edge middleware still terminates the request.

## Where the admin actually lives

Super-admin is at `admin.eatgooduganda.com` — a separate Vercel project, separate deployment, separate cookies, separate JWT secret. See `docs/01-ARCHITECTURE.md`.

`eatgooduganda.com/admin` is not an admin surface at all. It is a route that we deliberately make inaccessible.

## Implementation: Vercel edge middleware

Vercel runs `middleware.ts` at the edge _before_ the application code loads. We terminate the request there.

```ts
// apps/customer/middleware.ts
import { NextRequest, NextResponse } from 'next/server'

// Vercel's edge middleware signature works with Vite-on-Vercel too via
// the @vercel/edge adapter for Vite. We use the adapter's Request/Response.
export const config = {
  matcher: ['/admin', '/admin/:path*'],
}

export default function middleware(req: NextRequest) {
  return new NextResponse('Forbidden', {
    status: 403,
    headers: {
      'Content-Type': 'text/plain; charset=utf-8',
      'Cache-Control': 'public, max-age=3600, s-maxage=86400',
      'X-Robots-Tag': 'noindex, nofollow, nosnippet, noarchive',
      // Intentionally no Content-Security-Policy — there is no body HTML
    },
  })
}
```

If the customer app is Vite-based (which it is), we use `@vercel/edge` directly:

```ts
// apps/customer/api/edge-admin-403.ts (Vite + Vercel Edge function)
import type { VercelRequest, VercelResponse } from '@vercel/node'

export default function handler(_req: VercelRequest, res: VercelResponse) {
  res.setHeader('Content-Type', 'text/plain; charset=utf-8')
  res.setHeader('Cache-Control', 'public, max-age=3600, s-maxage=86400')
  res.setHeader('X-Robots-Tag', 'noindex, nofollow')
  res.status(403).send('Forbidden')
}

export const config = { runtime: 'edge' }
```

Routed via `vercel.json`:

```json
{
  "rewrites": [
    { "source": "/admin", "destination": "/api/edge-admin-403" },
    { "source": "/admin/(.*)", "destination": "/api/edge-admin-403" }
  ]
}
```

## Verification

A Playwright test confirms behaviour on every deploy:

```ts
// apps/customer/e2e/admin-403.spec.ts
import { test, expect } from '@playwright/test'

test.describe('/admin on customer host', () => {
  for (const path of ['/admin', '/admin/', '/admin/login', '/admin/deep/nested/path']) {
    test(`returns 403 for ${path}`, async ({ request }) => {
      const res = await request.get(path, { maxRedirects: 0 })
      expect(res.status()).toBe(403)
      expect(res.headers()['content-type']).toContain('text/plain')
      expect(await res.text()).toBe('Forbidden')
    })
  }
})
```

The test runs on every PR (preview deployment) and every production deploy.

## Security headers that still apply

Even though the 403 page has no HTML body, the response still carries our global security headers set on the Vercel project:

- `Strict-Transport-Security: max-age=63072000; includeSubDomains; preload`
- `X-Frame-Options: DENY`
- `X-Content-Type-Options: nosniff`
- `Referrer-Policy: strict-origin-when-cross-origin`

## What we did NOT consider acceptable

1. **Returning a React route.** A component like `<AdminAccessDenied />` renders a 200 response. No.
2. **Returning a 404.** 404 implies the path does not exist. It does — it exists as a forbidden path. 403 is the correct semantic.
3. **Redirecting to `admin.eatgooduganda.com`.** That would invite everyone to the real admin host, which we don't advertise publicly.
4. **Using `robots.txt` alone.** Scrapers don't care about robots.txt. A 403 at the edge is enforceable.
5. **Putting auth on a rendered page.** The page has nothing to authenticate against here — it simply must not exist.

## If we decide later to host the admin at `eatgooduganda.com/admin`

We would undo this middleware and mount the super-admin app at that path. That change would be documented in `docs/17-DECISIONS_LOG.md` and would require revisiting session cookie domains. Not in scope for v1.
